The Solution: Migrate

The Migrate module is a OO framework to enable complex migrations of content into Drupal.

It also has a fairly vertical learning curve.

Migrate Features

  • Repeatable imports
  • Partial imports
  • Migrate mapping
  • Useful methods. i.e. PrepareRow and Prepare
  • Highwater marks

Migrate Architecture

Consists of a number of classes that handle most of the Drupal heavy lifting

  • Migration
  • MigrateSource
  • MigrateDestination
  • MigrateMap
  • MigrateFieldMapping

Migration

Main class that any user-defined classes will be based off of.

MigrateSource

Class which handles understanding various types of migration sources

Sources Include:

  • SQL Query
  • XML
  • JSON
  • Other DBs: Oracle/MSSQL
  • You can write custom sources

MigrateDestination

Class which handles understanding various types of migration destinations within Drupal

MigrateMap

Consists of a number of classes that handle most of the Drupal heavy lifting

MigrateFieldMapping

Consists of a number of classes that handle most of the Drupal heavy lifting

The Approach

  • Pull data from Commonspot
  • Maintain mostly the same content types
  • 1:1 field mappings.
  • Migrate maintained relationships. i.e. authors, metadata

The Problem

  • No direct access to our MSSQL DB
  • Even developers have to go through Citrix
  • Wat do?

The Hack

  1. Export content as delimited file
  2. Import that into MySQL
  3. Run Migrate against that

Problems

  • Transfer problems with long text fields and non-standard characters
  • Takes a while to do all the steps
  • Not even close to real-time

The Better Solution

We are presently converting to a JSON feed of 90% of content.

  • Way fewer format issues feed is auto-escaped
  • near real-time import—When launch comes, less downtime on switch over
